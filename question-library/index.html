Product case study

A configurable question library for a survey platform

How I'd approach turning a fixed, 250-question survey library into an admin-configurable, AI-assisted system — from problem framing through interactive prototype.

RoleProduct / customer success leadership
ContextEnterprise employee experience (EX) survey platform
FocusProduct strategy, permissions design, applied AI
01 — the problem

A library that can't flex to the customer

On EX survey platforms, admin and creator users typically choose questions from a large fixed library — organized by category, often 200+ items — but usually cannot add, hide, or remove questions. Every user sees the same full library regardless of role.

Working closely with enterprise customers for years, I saw this cause two recurring problems: the library can't evolve to reflect a customer's own language or priorities, and creator users are exposed to hundreds of items even when only a narrow slice is relevant to their team, function, or seniority — creating noise and inconsistent survey design across an organization.

02 — what it should do

Core user stories

  • As an admin, I can add custom questions so the library reflects our own language and priorities.
  • As an admin, I can hide or remove questions so creators aren't shown irrelevant or outdated items.
  • As an admin, I can see which benchmark comparisons are available for a question, at the individual question level.
  • As an admin, I can control which portion of the library specific creator users or roles can see.
  • As an admin, when I save a new custom question, I get AI feedback on wording, duplication, and benchmark eligibility before it publishes.
  • As a user editing a question selected from the library, I'm warned if my changes are substantial enough that the question would lose its benchmark match, so I can decide whether to proceed as a new custom item or revert.
  • As a creator, I only see the questions I've been given access to, clearly organized by category.
03 — design decisions worth calling out

Where the judgment calls are

Hide, don't delete

Removing a question outright breaks historical survey data. Hiding is the default: it removes a question from future use while preserving what's already been measured.

Benchmarks live at the question level

Each question maps to its own benchmark. A highly customized question may have no benchmark match at all, and the system needs to say so plainly rather than implying a match that isn't there.

AI validation is a soft gate, not a hard block

When an admin saves a custom question, AI checks it for wording clarity, duplication against the existing library, and benchmark mapping. The admin sees the feedback and can still publish — the goal is an informed decision, not a rule enforced on their behalf.

Editing a library question can void its benchmark

Adding a new question checks whether it maps to a benchmark. Editing an existing one is a different, riskier check: has this question drifted far enough from its original wording that the existing benchmark no longer applies? If so, the user is warned before saving — the benchmark data already exists, so a silent mismatch here is worse than one on a brand-new question.

05 — scope

Phasing and open questions

Phase 1: guided hide/show and reorganization, per-question or per-group visibility controls, read-only benchmark status.

Phase 2: fully custom question authoring with AI validation at save time; admin control over which specific benchmark cuts are selectable per question.

Open questions: whether visibility needs individual-question granularity or category/group-level is sufficient for v1; how custom questions get validated for benchmarking eligibility over time; whether AI validation feedback should ever require an explicit override/justification rather than a simple "publish anyway."
